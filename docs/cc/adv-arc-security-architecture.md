# 보안 아키텍처 (ADV_ARC)

## AuthFusion SSO Server 보안 아키텍처 기술서

---

## 1. 개요

본 문서는 AuthFusion SSO Server의 보안 아키텍처를 기술한다.
TOE의 보안 도메인 분리, 자체 보호 메커니즘, 비우회성, 3층 구조를 설명한다.

### 1.1 아키텍처 설계 원칙

1. **보안 도메인 분리**: 보안 기능 간 명확한 경계 설정
2. **자체 보호 (Self-Protection)**: TOE 보안 메커니즘의 무결성 보호
3. **비우회성 (Non-Bypassability)**: 보안 기능을 우회할 수 없는 구조
4. **최소 권한 (Least Privilege)**: 각 구성 요소는 필요한 최소 권한만 보유
5. **심층 방어 (Defense in Depth)**: 다층 보안 메커니즘 적용

---

## 2. 3층 구조

### 2.1 구조 개요

AuthFusion Platform은 CC 평가를 효율적으로 진행하기 위해 3층 구조를 채택한다.

```
┌────────────────────────────────────────────────────────────────┐
│                      제1층: TOE 핵심 (Core)                     │
│                                                                 │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐               │
│  │ OIDC 인증   │ │ 토큰 관리   │ │ 세션 관리   │               │
│  │ 엔진        │ │ 엔진        │ │ 엔진        │               │
│  └─────────────┘ └─────────────┘ └─────────────┘               │
│  ┌─────────────┐ ┌─────────────┐ ┌─────────────┐               │
│  │ 암호키 관리 │ │ 감사 로그   │ │ 접근 제어   │               │
│  │             │ │ 엔진        │ │ 엔진        │               │
│  └─────────────┘ └─────────────┘ └─────────────┘               │
├────────────────────────────────────────────────────────────────┤
│                   제2층: TOE 에이전트 (Agent)                    │
│                                                                 │
│  ┌────────────────────────────────────────────┐                 │
│  │ SSO Agent (Servlet Filter)                 │                 │
│  │ - 토큰 검증 / 세션 연계 / 요청 필터링      │                 │
│  └────────────────────────────────────────────┘                 │
├────────────────────────────────────────────────────────────────┤
│                  제3층: 운영 환경 (비-TOE)                       │
│                                                                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐          │
│  │PostgreSQL│ │  Vault   │ │   NTP    │ │   SIEM   │          │
│  │   DB     │ │  / HSM   │ │  Server  │ │          │          │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘          │
└────────────────────────────────────────────────────────────────┘
```

### 2.2 계층별 역할

#### 제1층: TOE 핵심 (SSO Server)

| 구성 요소 | 역할 | 보안 기능 |
|-----------|------|----------|
| OIDC 인증 엔진 | Authorization Code + PKCE 흐름 처리 | FIA_UAU.1, FIA_UID.1 |
| 토큰 관리 엔진 | JWT 토큰 생성/검증/폐기 | FCS_COP.1 |
| 세션 관리 엔진 | SSO 세션 생명주기 관리 | FTA_SSL.3 |
| 암호키 관리 | RSA 키 페어 생성/로테이션/파기 | FCS_CKM.1, FCS_CKM.4 |
| 감사 로그 엔진 | 보안 이벤트 기록/조회 | FAU_GEN.1, FAU_SAR.1 |
| 접근 제어 엔진 | RBAC 기반 정책 시행 | FDP_ACC.1, FDP_ACF.1 |

#### 제2층: TOE 에이전트 (SSO Agent)

| 구성 요소 | 역할 | 보안 기능 |
|-----------|------|----------|
| Servlet Filter | HTTP 요청 인터셉트 및 인증 검사 | FIA_UAU.1 |
| 토큰 검증기 | JWT 토큰 서명 검증 (공개키 사용) | FCS_COP.1 |
| 세션 연계 | SSO Server 세션과 로컬 세션 동기화 | FTA_SSL.3 |
| SSO Server 클라이언트 | SSO Server API 통신 | FTP_TRP.1 |

#### 제3층: 운영 환경 (비-TOE)

| 구성 요소 | 역할 | TOE 의존성 |
|-----------|------|-----------|
| PostgreSQL | 데이터 영속화 | TOE가 JDBC로 접근 |
| Vault/HSM | 마스터 키 보관 | TOE가 API/PKCS#11로 접근 |
| NTP | 시간 동기화 | OS 레벨 동기화 |
| SIEM | 감사 로그 장기 보관 | TOE가 로그 전송 |

## 3. 보안 도메인 분리

### 3.1 도메인 정의

TOE는 다음과 같은 보안 도메인으로 분리된다:

```
┌────────────────────────────────────────────────────────┐
│                    SSO Server JVM                       │
│                                                         │
│  ┌───────────────────────────────────────────────┐      │
│  │        Spring Security Filter Chain            │      │
│  │  (모든 HTTP 요청에 대한 보안 정책 시행)          │      │
│  └───────────┬───────────────────────┬────────────┘      │
│              │                       │                    │
│  ┌───────────┴──────┐   ┌───────────┴──────┐            │
│  │  인증 도메인      │   │  인가 도메인      │            │
│  │                   │   │                   │            │
│  │  - AuthService    │   │  - SecurityConfig │            │
│  │  - OIDC Endpoints │   │  - RBAC Engine    │            │
│  │  - BruteForce     │   │  - ScopeService   │            │
│  │    Protection     │   │                   │            │
│  └───────────────────┘   └───────────────────┘            │
│              │                       │                    │
│  ┌───────────┴──────┐   ┌───────────┴──────┐            │
│  │  암호 도메인      │   │  감사 도메인      │            │
│  │                   │   │                   │            │
│  │  - KeyPairManager │   │  - AuditService   │            │
│  │  - JwtProvider    │   │  - AuditEvent     │            │
│  │  - KeyEncryption  │   │    Repository     │            │
│  │  - PasswordHash   │   │                   │            │
│  └───────────────────┘   └───────────────────┘            │
│              │                                            │
│  ┌───────────┴──────┐                                    │
│  │  세션 도메인      │                                    │
│  │                   │                                    │
│  │  - SessionService │                                    │
│  │  - SessionStore   │                                    │
│  └───────────────────┘                                    │
└────────────────────────────────────────────────────────┘
```

### 3.2 도메인 간 상호작용

| 호출자 | 피호출자 | 인터페이스 | 설명 |
|--------|---------|-----------|------|
| 인증 도메인 | 암호 도메인 | PasswordHashService | 비밀번호 해시 비교 |
| 인증 도메인 | 세션 도메인 | SessionService | 세션 생성/검증 |
| 인증 도메인 | 감사 도메인 | AuditService | 인증 이벤트 기록 |
| 인증 도메인 | 암호 도메인 | JwtTokenProvider | 토큰 생성 |
| 인가 도메인 | 암호 도메인 | JwtTokenParser | 토큰 검증 |
| 인가 도메인 | 감사 도메인 | AuditService | 접근 제어 이벤트 기록 |
| 암호 도메인 | 감사 도메인 | AuditService | 키 관리 이벤트 기록 |

### 3.3 도메인 격리 메커니즘

- **패키지 수준 격리**: 각 보안 도메인은 별도의 Java 패키지로 구성
- **인터페이스 기반 통신**: 도메인 간 통신은 정의된 인터페이스를 통해서만 수행
- **의존성 주입**: Spring IoC 컨테이너를 통한 구성 요소 관리
- **접근 제한자**: 도메인 내부 구현 클래스는 `package-private` 또는 `private` 접근 제한

## 4. 자체 보호 (Self-Protection)

### 4.1 TSF 데이터 보호

TOE는 다음 메커니즘으로 자체 보안 데이터를 보호한다:

#### 4.1.1 서명키 보호

- 개인키는 AES-256-GCM으로 암호화되어 DB에 저장
- 마스터 키는 Vault/HSM 등 외부 보안 모듈에서 관리
- 메모리 내 키 데이터는 사용 후 즉시 클리어

#### 4.1.2 비밀번호 보호

- 비밀번호는 BCrypt(cost factor 12+)로 해싱하여 저장
- 평문 비밀번호는 메모리에 최소 시간만 유지
- 비밀번호 이력 관리로 재사용 방지

#### 4.1.3 세션 데이터 보호

- 세션 ID는 암호학적으로 안전한 랜덤 생성기(SecureRandom) 사용
- 세션 저장소는 JVM 내부 메모리로 외부 접근 차단
- 세션 데이터는 직렬화/역직렬화 시 민감 정보 필터링

#### 4.1.4 감사 로그 보호

- 감사 레코드는 DB에 저장 시 HMAC 무결성 검증 값 포함
- CC 모드에서 감사 테이블 UPDATE/DELETE 제한
- 감사 로그 생성 모듈은 다른 모듈에서 조작 불가

### 4.2 실행 환경 보호

- **JVM 보안 매니저**: 파일 시스템/네트워크 접근 제한 (선택)
- **systemd 강화**: `NoNewPrivileges`, `ProtectSystem` 등 적용
- **최소 권한 실행**: 전용 시스템 사용자(authfusion)로 실행
- **읽기 전용 배포**: JAR 파일 실행 권한만 부여 (쓰기 불가)

## 5. 비우회성 (Non-Bypassability)

### 5.1 요청 처리 아키텍처

모든 HTTP 요청은 반드시 Spring Security Filter Chain을 거치도록 설계된다:

```
클라이언트 요청
     │
     ▼
┌─────────────────────────────────┐
│  RateLimitFilter                │  ← 요청 빈도 제한
├─────────────────────────────────┤
│  CorsFilter                    │  ← CORS 정책 검사
├─────────────────────────────────┤
│  SecurityFilterChain            │  ← 인증/인가 검사
│  - BearerTokenFilter           │     토큰 검증
│  - SessionValidationFilter     │     세션 유효성 검사
│  - AuthorizationFilter         │     접근 권한 검사
├─────────────────────────────────┤
│  AuditFilter                   │  ← 감사 로그 기록
├─────────────────────────────────┤
│  Controller (비즈니스 로직)      │  ← 요청 처리
└─────────────────────────────────┘
```

### 5.2 비우회성 보장 메커니즘

#### 5.2.1 필터 체인 강제

- Spring Security `FilterChainProxy`는 모든 요청에 대해 자동 적용
- 필터 순서는 `@Order` 어노테이션으로 고정되어 변경 불가
- 공개 엔드포인트(JWKS, Discovery)도 필터 체인을 통과 (인증만 면제)

#### 5.2.2 인증 강제

- 인증이 필요한 모든 API는 `SecurityConfig`에서 명시적으로 보호
- `authenticated()` 또는 `hasRole()` 설정이 없는 경우 기본적으로 접근 거부
- OIDC 엔드포인트는 프로토콜 수준에서 클라이언트/사용자 인증 수행

#### 5.2.3 감사 강제

- `AuditService`는 보안 이벤트 발생 지점에서 동기적으로 호출
- 감사 로그 기록 실패 시에도 원래 작업은 완료되지만 로그 오류가 별도 기록
- CC 모드에서는 감사 로그 저장 실패 시 해당 요청을 거부하는 옵션 제공

#### 5.2.4 SSO Agent 비우회성

- SSO Agent는 Jakarta Servlet Filter로 구현되어 모든 요청을 인터셉트
- `web.xml` 또는 `FilterRegistration`에서 `/*` 패턴으로 등록
- 필터 우선순위가 가장 높아 다른 필터보다 먼저 실행

## 6. 보안 기능 상호작용

### 6.1 인증 흐름에서의 보안 기능 연계

```
사용자 로그인 요청
     │
     ├── [FIA_AFL.1] BruteForceProtection: 계정 잠금 여부 확인
     │
     ├── [FIA_UID.1] UserService: 사용자 식별
     │
     ├── [FIA_UAU.1] AuthService: 자격증명 검증
     │       │
     │       └── [FCS_COP.1] PasswordHashService: BCrypt 해시 비교
     │
     ├── [FIA_USB.1] SecurityContext: 사용자-주체 바인딩
     │
     ├── [FCS_COP.1] JwtTokenProvider: JWT 토큰 생성 (RS256 서명)
     │       │
     │       └── [FCS_CKM.1] KeyPairManager: 서명 키 페어 조회
     │
     ├── [FTA_SSL.3] SessionService: SSO 세션 생성
     │
     └── [FAU_GEN.1] AuditService: 감사 이벤트 기록
```

### 6.2 토큰 검증 흐름에서의 보안 기능 연계

```
API 요청 (Bearer Token)
     │
     ├── [FCS_COP.1] JwtTokenParser: JWT 서명 검증
     │       │
     │       └── [FCS_CKM.2] JwkProvider: 공개키 조회
     │
     ├── [FDP_ACC.1] SecurityConfig: 접근 권한 확인
     │       │
     │       └── [FDP_ACF.1] RBAC: 역할 기반 접근 제어
     │
     ├── [FTA_SSL.3] SessionService: 세션 유효성 확인
     │
     └── [FAU_GEN.1] AuditService: 접근 감사 기록
```

## 7. 초기화 및 안전한 시작

### 7.1 TOE 초기화 절차

TOE는 다음 순서로 초기화되며, 각 단계가 성공해야 다음 단계로 진행된다:

1. **설정 로딩**: `application-cc.yml` 프로파일 로딩
2. **CC 모드 검증**: CC 모드 필수 파라미터 확인
3. **DB 연결 확인**: PostgreSQL 연결 및 스키마 마이그레이션
4. **마스터 키 로딩**: Vault/HSM에서 마스터 키 획득
5. **서명키 초기화**: 활성 서명키 존재 여부 확인 (없으면 생성)
6. **보안 필터 체인 구성**: Spring Security 필터 체인 초기화
7. **감사 모듈 초기화**: AuditService 준비 상태 확인
8. **서비스 시작**: HTTP 리스너 활성화

### 7.2 안전한 시작 보장

- 초기화 과정에서 오류 발생 시 TOE는 시작되지 않음
- CC 모드 필수 설정이 누락된 경우 명확한 오류 메시지와 함께 종료
- 시작 시 자체 진단(Self-Test)을 수행하여 암호 모듈 정상 동작 확인

## 8. 장애 복구 및 안전한 상태

### 8.1 안전한 실패 (Fail-Secure)

TOE는 장애 발생 시 안전한 상태를 유지한다:

| 장애 상황 | 대응 동작 |
|----------|----------|
| DB 연결 실패 | 인증/인가 요청 거부 (기존 세션은 메모리에서 유지) |
| Vault 연결 실패 | 새 키 생성 불가, 기존 키로 계속 운영 |
| 메모리 부족 | 새 세션 생성 거부, 기존 세션 유지 |
| 디스크 풀 | 감사 로그 경고 후 요청 거부 (CC 모드) |

### 8.2 복구 절차

장애 복구 후 반드시 다음 사항을 확인한다:

1. CC 모드 활성화 상태 확인
2. 감사 로그 연속성 확인 (장애 전후 간격 기록)
3. 활성 세션 상태 검증
4. 서명키 유효성 확인
